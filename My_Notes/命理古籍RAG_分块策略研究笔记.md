# 命理古籍 RAG 分块策略研究笔记

> 整理日期：2026-05-25  
> 场景：命理古籍（文言文、理论+案例混排）的 RAG 系统构建  
> 参考领域：中医RAG、法律RAG、领域专用RAG

---

## 一、任务特殊性分析

命理古籍与通用文档相比，有以下"硬核"特点：

| 特点 | 带来的挑战 |
|------|-----------|
| 文言文，极度精炼 | 5个字可能等于一套完整理论，固定token切割会导致语义残缺 |
| 理论 + 案例混排 | 同一本书前半是干支理论，后半是命例解析，检索需求完全不同 |
| 注疏体结构 | 原经文 + 注解是语义整体，必须绑定在一起检索 |
| 隐喻和专有名词密集 | "金水伤官"、"七杀透干"等，通用 embedding 模型无法正确理解语义 |
| 表格/对照矩阵大量存在 | 纳音表、神煞表、旺衰表等，传统 chunk 方式会破坏表格结构 |

## 二、RAG 分块策略全景（通用）

### 2.1 六大类分块策略

#### 第一类：机械式分块
| 策略 | 原理 | 优点 | 缺点 | 适合场景 |
|------|------|------|------|---------|
| Fixed-size | 按 token 数硬切 | 极简、速度快 | 可能截断语义 | 格式均匀的日志、表格 |
| Sliding Window | 固定大小 + overlap | 减少边界信息丢失 | 存储冗余 | 连续性强的长文本 |

#### 第二类：结构感知分块
| 策略 | 原理 | 优点 | 缺点 | 适合场景 |
|------|------|------|------|---------|
| Sentence-based | 按句号/换行切 | 语义完整 | chunk 大小不可控 | 新闻、FAQ、对话 |
| Paragraph-based | 按段落切 | 保留段落逻辑单元 | 段落长短悬殊 | 学术论文、技术文档 |
| Recursive Character Splitting | 段落→句子→词层级降级 | 兼顾大小与语义，工程性价比最高 | — | 通用场景，大多数 RAG 的默认起点 |
| Document Structure-aware | 利用 Markdown/HTML 标题等显式结构 | chunk 天然对齐文档层级 | 依赖文档格式规范 | 技术手册、知识库文档 |

#### 第三类：语义感知分块
| 策略 | 原理 | 优点 | 缺点 | 适合场景 |
|------|------|------|------|---------|
| Semantic Chunking | 计算相邻句 embedding 余弦相似度，相似度突降处切割 | 语义内聚，召回率提升 9–15% | 计算成本高 3–5x | 多话题混合文档，生产级 RAG |
| Proposition-based | 用 LLM 将文本改写为原子事实陈述 | 检索精准度最高 | LLM 处理成本高 | 医疗、法律、金融等知识密集型问答 |

#### 第四类：层次化分块
| 策略 | 原理 | 优点 | 缺点 | 适合场景 |
|------|------|------|------|---------|
| Small-to-Big（Parent-Child） | 小 chunk 用于检索，返回父 chunk 给 LLM | 兼顾检索精度与上下文完整性 | 索引设计复杂 | 长文档问答，目前工程最流行 |
| RAPTOR | 递归摘要构建层次树，多粒度召回 | 能回答全局性问题 | 构建成本高 | 长篇研报、技术综述 |
| HiChunk | 层次分块框架，自动合并 | 中文文档证据召回 74%→81%，FinanceBench 精度 +22% | — | 企业知识库、中文专业文档 |

#### 第五类：LLM 驱动分块
| 策略 | 原理 | 优点 | 缺点 | 适合场景 |
|------|------|------|------|---------|
| Agentic Chunking | 让 LLM 直接判断语义边界 | 最灵活，效果理论最优 | 成本极高，速度慢 | 高价值小批量文档（合同、核心知识） |
| Meta-Chunking | 基于逻辑感知的 LLM 语义分割 | 深度语义理解 | — | 精炼密集文本 |
| LumberChunker | 用 Qwen2.5-14B 等模型预测最优分割点 | 针对中文效果好 | 成本较高 | 高精度中文文本分割 |

#### 第六类：新兴方法
| 策略 | 原理 | 优点 | 缺点 | 适合场景 |
|------|------|------|------|---------|
| Late Chunking | 先整文编码获得全局 embedding，再按位置切 chunk | 每个 chunk 携带全文上下文，解决跨 chunk 指代问题 | 需要支持长上下文的 embedding 模型 | 指代关系复杂的文档（论文、法律文本） |
| Table-Aware Row-Level RAG | 将表格每一行作为独立语义单元向量化 | 解决传统 RAG 把整表编成一个向量的问题，行级精准检索 | — | 古籍表格、医学对照表、法规条目（2026年新论文） |
| Summary-Augmented Chunking (SAC) | 嵌入前为每块追加文档级全局摘要 | 减少"文档级检索不匹配"，法律/科学领域有效 | — | 法律文档、学术论文 |

## 三、命理古籍 RAG 专项方案

### 3.1 内容分层预处理（核心思路）

命理古籍在分块之前，先按内容类型分层并打标签：

```
📚 命理古籍
  ├── 【理论层】五行生克、十神、格局理论
  │     → 小 chunk（128–256 tokens），用于精确概念检索
  │
  ├── 【命例层】真实命造案例解析
  │     → 中 chunk（512–1024 tokens），保持案例完整性
  │
  └── 【表格层】神煞对照表、纳音五行表、旺衰速查表
        → 行级分块（Table-Aware Row-Level RAG）
```

**为什么要分层：**  
用户问"什么是七杀格"（→ 理论检索）和"帮我分析这个命盘"（→ 命例检索）是完全不同的检索需求，混在一起检索会互相污染，降低准确率。

---

### 3.2 针对每层的分块策略

#### 理论层：标点符号驱动 + 注疏绑定

- 以中文全角标点（。！？；）为边界，而非固定 token 数
- **注疏体必须绑定**：经文 + 注解作为一个 chunk，不可分割

```
示例：
《滴天髓》原文："五阳皆阳丙为最，五阴皆阴癸为至"
注解："丙火为阳中之阳，其性刚烈..."
→ 上述两句必须在同一个 chunk，不能因为超过 token 限制被切开
```

- 推荐大小：128–256 tokens
- 对应策略：Semantic Chunking（计算句间相似度，相似度突降处切）

#### 命例层：Small-to-Big（父子结构）

```
父 chunk（整个命例，约 800 tokens）→ 返回给 LLM 生成，保证上下文完整
  ├── 子 chunk1（命盘基本信息）→ 用于向量检索
  ├── 子 chunk2（原局分析）   → 用于向量检索
  └── 子 chunk3（运势评断）   → 用于向量检索
```

检索时用子 chunk 定位，返回父 chunk 给模型。

#### 表格层：Row-Level RAG

命理古籍中大量使用对照表，每一行单独向量化：

```
纳音表第3行: "丙寅、丁卯 → 炉中火"  ← 独立成一个 chunk
纳音表第4行: "戊辰、己巳 → 大林木"  ← 独立成一个 chunk
```

这样"丙寅日主是什么纳音"才能精确命中，不会把整张表混为一坨。

---

### 3.3 元数据富化

每个 chunk 除了向量，还需附加元数据：

```json
{
  "book": "滴天髓",
  "chapter": "卷一·天干论",
  "type": "theory",          // theory | case | table
  "case_id": null,           // 命例层才有值，用于父子关联
  "row_index": null          // 表格层才有值
}
```

元数据在检索时可以做**过滤**（如"只在命例层里找"），也可以作为重排的权重依据。

---

### 3.4 推荐 Embedding 模型

| 优先级 | 模型 | 理由 |
|--------|------|------|
| ⭐⭐⭐ 首选 | **BGE-M3**（BAAI） | 支持 8192 tokens；密集+稀疏+多向量三模式；中文 SOTA；免费开源，工程成熟 |
| ⭐⭐ 备选 | **TongGu-7B**（华南理工） | 首个文言文专用模型；内置标点恢复+命名实体识别；对古文理解优于通用模型；但 7B 参数部署成本高 |
| ⭐ 轻量 | **bge-base-zh-v1.5** | 资源受限时使用，中文优化，轻量级 |

**工程建议**：起步用 BGE-M3，如果测评后召回质量不够理想，再考虑引入 TongGu 或对 BGE 做古文语料微调。

---

### 3.5 整体架构

```
命理古籍原文（PDF / TXT / 扫描件）
    ↓
[OCR + 文本清洗]（古籍扫描建议用专用 OCR 引擎，如 kandianguji.com）
    ↓
[内容分类] 自动打标签：理论 / 命例 / 表格
    ↓
[差异化分块]
    理论层 → 标点驱动 + 注疏绑定（128–256 tokens）
    命例层 → Small-to-Big 父子结构（子 128，父 800）
    表格层 → Row-Level 行级分块
    ↓
[元数据富化] book / chapter / type / case_id / row_index
    ↓
[BGE-M3 向量化]
    ↓
[向量数据库] Chroma（本地开发）/ Milvus（生产）
    ↓
[检索] 向量相似度召回 → Reranker 重排
    ↓
[LLM 生成回答]
```

## 五、关键数据点（可用于项目汇报/简历包装）

- 语义分块 vs 固定大小分块：精炼密集文本召回率提升 **9–15%**，但计算成本高 **3–5x**
- 递归分块 vs 最优定长分块：域加权精度提升 **45%**
- Small-to-Big（HiChunk）：证据召回率提升 **7%**，FinanceBench 答案精度提升 **22%**
- 医疗自适应分块：逻辑主题边界 vs 固定大小 = **87% vs 13%** 准确率
- 语义相似度方法（Chroma 研究）：400 token chunk 达 **91–92% 召回率**

## 六、工程实施路线（三阶段）

### 阶段一：快速验证（Week 1–2）
1. 用 RecursiveCharacterTextSplitter（400 token，20% overlap）跑基线
2. 用 BGE-M3 做向量化
3. 搭 RAGAS 评测流水线，记录 Recall@K 和 Answer Accuracy

### 阶段二：优化分块（Week 3–4）
1. 实现内容分类器（理论/命例/表格自动打标签）
2. 对命例层引入 Small-to-Big 父子结构
3. 对表格层实现 Row-Level 分块
4. 对比评测，量化提升幅度

### 阶段三：高阶优化（Month 2+）
1. 引入 Reranker（如 bge-reranker-v2-m3）
2. 探索 TongGu 模型在古文理解上的提升效果
3. 考虑 GraphRAG：将命理实体（日主、十神、格局）构建知识图谱，提升关系型检索
