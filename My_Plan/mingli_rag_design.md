# 命理测算 RAG 项目设计笔记

> 创建于 2026-05-22
> 背景：北邮研二，目标 Agent 开发秋招，项目一选择命理 RAG

---

## 项目定位

**解决的痛点：**
- 市面 APP 排盘免费但分析要收费
- DeepSeek/豆包等通用大模型四柱排盘不准（根基错 → 全错）
- 通用大模型缺乏命理专业知识深度

**核心知识库来源：**
- 古籍：穷通宝典、三命通会、滴天髓、渊海子平、千里命稿、子平真诠、神峰通考
- 名人案例库：已知八字 + 已知人生轨迹（用于评估和 few-shot）

---

## 整体架构流程

```
用户输入生辰八字 + 问题
        ↓
① Tool Calling → 调排盘库 → 获取准确四柱
        ↓
② 意图识别 → 判断问题类型（感情/事业/本命/流年...）
        ↓
③ 查询重写 → 口语表达 → 命理专业术语
        ↓
④ 混合检索 → BM25 + 向量双路召回 → RRF 融合
        ↓  （检索两层存储的古籍库 + 名人案例库）
⑤ Reranker → 精排 Top-K 结果
        ↓
⑥ LLM 生成回答（携带八字 + 古籍原文+注解 + 相关案例）
        ↓
⑦ 会话记忆 → 保存本轮结果供下轮使用
        ↓
⑧ RAGAS 离线评估（用名人案例测试集定期跑分）
```

## 八大设计点详解

### ① Tool Calling 排盘（独有，其他简历无）

四柱排盘是纯确定性算法，用开源库本地跑即可，无需爬虫或 API。

```python
# 推荐库：lunar-python 或 bazi
def get_bazi(year, month, day, hour, gender) -> dict:
    # 返回：年柱、月柱、日柱、时柱、大运起运...
    pass
```

**Agent 流程：** 先调此工具获取准确八字 → 再基于八字进行检索和分析

**面试亮点：** 工具调用 + RAG 的组合，比单纯 RAG 技术层次高；且解决了通用大模型排盘不准的根本问题。

---

### ② 意图识别树

命理问题天然分类，不同类型走不同检索策略：

```
命理问题
├── 本命分析（性格、天赋、格局）
├── 大运流年（某年/某段运势）
├── 专项咨询
│   ├── 感情婚姻 → 重点检索"桃花、夫妻宫、官杀"
│   ├── 事业财运 → 重点检索"官杀、财星、食伤"
│   └── 健康      → 重点检索"身体部位对应五行"
└── 排盘解释（某个柱/神是什么意思）
```

**实现方式：** 用 LLM 做分类器，结果缓存在 Redis，置信度不足时反问用户澄清。

**参考来源：** 舍友 Ragent 项目的意图识别模块 + 简历8 的 Router 架构

---

### ③ 查询重写

**问题：** 用户口语化表达与古籍术语不匹配

```
用户问：  "我今年能发财吗"
↓ LLM 重写
检索词：  "日元与财星关系、流年财星透出、伤官生财"
```

**复杂问题拆分：** "我的感情和事业今年怎么样" → 拆成两个子问题并行检索

**兜底策略：** 重写失败时保留原始问题继续检索

---

### ④ BM25 + 向量混合检索 + RRF 融合（核心）

**为什么命理场景必须用混合检索：**
- "甲木"和"乙木"差一字，含义天壤之别 → 纯向量检索会混淆
- 古籍用词极度精确，BM25 能抓住精确字符
- 两路互补，召回率显著提升

**RRF 融合公式：**
```
score(d) = Σ 1 / (k + rank_i(d))
# k=60（经验值），rank_i 是第 i 路检索中文档 d 的排名
```

**参考来源：** 简历 13、14、7、2 均有此设计

---

### ⑤ Reranker 重排序

检索出 Top-20 → Cross-Encoder 精排 → 取 Top-5 给 LLM

**推荐模型：** BGE-Reranker-v2（BAAI 出品，中文效果好）

**命理场景价值：** 初步检索可能召回所有"七杀"相关段落，Reranker 能把"七杀制化"精准排到最前面。

**参考来源：** 简历 2、14、15

---

### ⑥ 两层文档存储（独有设计）

古籍文言文语义压缩，直接 embedding 效果差。解决方案：

```
存储结构：
{
  "原文":   "官印相生，贵格也",
  "注解":   "当命局中正官与印绶同现且相互生助时，
             构成贵气格局。适用条件：日元需有根，
             官星不混杂，印星不被财克...",
  "出处":   "《滴天髓》格局篇",
  "标签":   ["格局", "官印", "贵格"]
}

检索：走注解层（语义清晰，embedding 效果好）
返回：带原文 + 注解（LLM 获得原典依据 + 现代解释）
```

**面试亮点：** 针对文言文特性的专项 RAG 工程优化，有明确的技术 why。

---

### ⑦ 会话记忆管理

命理咨询天然多轮：第一轮报八字，后续轮次问不同方面。

**策略：**
- 滑动窗口：保留最近 N 轮（N=5~10）
- 超过阈值：LLM 自动压缩摘要持久化
- 关键信息（八字本身）永久保留，不参与压缩

**参考来源：** Ragent 项目的会话记忆模块

---

### ⑧ RAGAS 评估体系（差异化最大的点）

整批 19 份简历中只有 1 人做了 RAGAS，是最稀缺的差异化点。

**测试集构建：**
- 20~50 个名人八字（毛泽东、李嘉诚、马云等，公开资料）
- 参考答案：命理 APP 专业输出 / 知名命理师分析
- 评估维度：检索相关性、回答忠实度、回答完整性

**RAGAS 四大指标：**
| 指标 | 含义 |
|------|------|
| Context Precision | 检索到的文段是否真的相关 |
| Context Recall | 相关文段有没有被检索到 |
| Faithfulness | 回答有没有产生幻觉 |
| Answer Relevancy | 回答是否切题 |

## 技术栈选型

| 层次 | 选型 | 理由 |
|------|------|------|
| 编排框架 | LangGraph | 支持有状态的 Agent 流程，意图路由用 Graph 节点表达 |
| 向量库 | ChromaDB（开发）/ Milvus（生产） | ChromaDB 轻量易上手，Milvus 生产级 |
| 稠密检索 | BGE-M3 embedding | 中文效果最好的开源 embedding 模型之一 |
| 稀疏检索 | BM25（rank_bm25 库） | 轻量，无需额外服务 |
| Reranker | BGE-Reranker-v2 | 与 BGE-M3 同系列，配合效果好 |
| 后端 | FastAPI | 标准选择，轻量 |
| 评估 | RAGAS | 自动化 RAG 评估框架 |
| 排盘 | lunar-python | 开源，本地运行，确定性算法 |
| 追踪 | LangSmith | Agent 调试和链路追踪 |
